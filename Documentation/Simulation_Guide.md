# Simulation Guide

This document describes how to execute Verilog testbenches, analyze waveforms, and debug functional errors.

## Running Simulations
Simulations model hardware behavior combinationally and sequentially without deploying designs onto physical FPGA boards. Vivado Simulator translates Verilog testbenches into executable waveforms.
- **Design Under Test (DUT)**: The target Verilog module being verified.
- **Testbench (TB)**: A wrapper module containing no physical pin inputs/outputs. It drives simulated stimulus inputs into the DUT and observes outputs.

## Understanding Waveforms
In the simulation waveform viewer:
- **Time Axes**: Run from left to right in nanoseconds (`ns`) or picoseconds (`ps`).
- **Binary Waves**: Represent logic state transitions between `0` (low) and `1` (high).
- **Bus Hex values**: Multiple signal lines (busses) display aggregate hexadecimal numbers inside parallel bars.
- **Undefined States (X)**: Red lines indicate uninitialized memory registers or driver contention.
- **High Impedance (Z)**: Blue lines represent disconnected inputs or tri-stated outputs (e.g. in SRAM/I2C buses).

## Using the Testbench
Testbenches utilize the following behavioral constructs:
- **`initial` Blocks**: Execute once at simulation startup to initialize input values.
- **`always` Blocks**: Repeat throughout execution (used extensively to generate clock cycles).
- **Time delays (`#10`)**: Direct the simulator to wait 10 time units before executing the next instruction line.
- **`$monitor`**: System task outputting changes in tracked registers to the console log stream automatically.

## Common Simulation Errors
- **Infinite Loops**: Caused by missing delays inside clock generation loops or state transition branches.
- **Hi-Z / High Impedance Output**: Caused by unassigned ports or incorrect bidirectional control settings.
- **Multi-Source Contention**: Caused by driving the same wire output register from two independent `always` blocks.

## Tips for Debugging
1. Look at console outputs generated by `$monitor` or `$display`.
2. Trace the clock edge updates to verify data shifts on the exact rising edge.
3. Verify that asynchronous active-low reset inputs (`rst_n`) are set high after initialization delays.
